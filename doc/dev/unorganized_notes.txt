  Graphics is pixel-based and palette-based. Internally we keep 320x200 graphics::legacy_screen which in essense is just a char[]  
When time comes (sdlhelper.cpp -> av_sync() -> SDL_SetColors()), said chars are used simply as indicies into a palette, where actual colors are stored
Sometimes said palette changes or gets mixed - there are some bugs due to that

  Code style is easy to proclaim, harder to adhere, and impossible not to nitpick.
 That's why it's usually called "guidelines", not rules
 But consistent "look" of the code does help
 
 currently code either follows or slowly converted to follow these directions:
 1) 4 spaces, no tabs
    too late to change and almost invisible to diagnose by eye
    there's a lot of code that spills to multiple lines, so needs spaces for staying neat
    between mixed and spaces-only latter one is easier to type
    
 2) `if() {`, `} else {` and other expressions of curcly brace on same line
    probably feels nice to not waste lines for once in the already-wasteful code?
    tho personally this is the easiest guideline to break, as some code needs breathing room between condition and code
    function signature is always on the different line, tho
    ```
    void foo()
    {
    ```
    
 3) variables should be declared as late as possible
    this is battle against C style, where `goto` can't jump over variable declarations
    we don't use goto, so we can declare variables late to easily see where and in which scope they are used
    
 4) ```
    if (AAAAA
        || BBBBB)
    ```
    having OR and AND on new line helps with rule 2
    
 5) `if(! A)`, `if(!A && !B)`
    it looks neat and attracts the eye to most important bits
    
 6) `type* var`, `type& var` - stick ymbol-modifier to the type
    general C++ style. Types are expressive, variables are dumb
    since variables are declared and assigned mostly at the same time, you don't encounter the need for `type *a, *b` much
